1. Executive Summary

CommandDeck is a self-hosted agentic software development platform. A user sends a Slack message describing a task, and a crew of 11 specialized AI agents decomposes the task into phased objectives, implements them in parallel git worktree branches, runs tests, reviews for security and quality via mandatory specialist reviews, creates a PR with structured evidence bundles, deploys a preview environment via GitHub Actions, and waits for human approval to merge.

The system runs as a single Docker container on an EC2 instance at 100.29.233.41. It combines a Slack bot (via Bolt Socket Mode), an HTTP dashboard with JSON API, and the mission orchestration engine into one Node.js process (server.js, 1463 lines). State is split across two SQLite databases (app.db for application state, telemetry.db for observability) plus filesystem artifacts for evidence bundles, briefings, and captain's logs.

Agents are spawned as claude -p subprocesses, each running in an isolated git worktree. The system currently manages 5 onboarded projects (WeatherAggregator, FurEverCare, eve, SharkShark, Skip Gastown) with the full lifecycle operational end-to-end.

Key numbers: 7,528 lines of JavaScript across 18 modules. 11 agent identity files. 2 hook scripts. 7 default standards, 10 crew preference files, 1 playbook. 218 passing tests. Two SQLite databases. One Docker container. Zero external dependencies beyond Claude Code CLI and GitHub CLI.

Figure 1 — Full system architecture showing all components in a single process

Single-process design: Everything runs in one Node.js process. The Slack bot uses Socket Mode (outbound WebSocket, no inbound ports required). The HTTP server on port 3000 serves the dashboard, JSON API, and SSE telemetry stream. The health patrol runs on a setInterval. This simplifies deployment but means a crash takes down all capabilities simultaneously.

Meta-Channel Resolution

The meta-channel (where project creation/onboarding is allowed) is auto-detected at startup from the bot's username. resolveMetaChannel() calls auth.test() to get the bot name, replaces underscores with hyphens, and searches Slack channels for a match.     |       |

    |       |

    |       |         1. Create worktree on branch commanddeck/{mission}/{obj.id}
    |       |         2. Spawn claude -p worker with agent identity + prompt
    |       |         3. Wait for all workers (Promise.all)
    |       |         4. Read evidence bundles, detect risk flags from files
    |       |         5. Merge Picard flags + detected flags (deduplicated)
    |       |         6. Update objective status: done | failed

    |       |       git merge {branch} --no-edit

Figure 5 — Complete mission lifecycle method flow

Follow-Up Missions

Mission.followUp() creates a new mission that reuses the parent's integration branch and PR. Workers branch from the integration branch (not main), so they see all previous work. After completion, the existing PR body is updated rather than creating a new PR.

Plan Revision

While a mission is pending_approval, thread replies trigger mission.revisePlan(feedback). This re-invokes Picard with the previous plan and user feedback, replaces work_items, and re-posts the plan for approval.

5. Worker System

lib/worker.js (410 lines) is the bridge between the orchestrator and the Claude Code CLI. Every agent runs as a claude -p subprocess.

Worker Execution Model

  worker.execute(worktreePath, objective, mission, { model })
    |
    +---> loadAgentIdentity(objective.assigned_to)
    |       Read agents/{name}.md
    |       parseFrontmatter() -> { identity, tools, model }
    |
    +---> Build prompt:
    |       [agent identity markdown]
    |       Executing objective {id}: {title}
    |       Description: {description}
    |       Mission state directory: {stateDir}
    |       When done: commit, write evidence, write briefing
    |
    +---> Determine model (precedence):
    |       1. project config.json model_overrides.{agent}
    |       2. agents/{name}.md frontmatter model
    |       3. COMMANDDECK_MODEL env var
    |       4. "claude-opus-4-6" (hardcoded default)

    |       '-p', prompt,
    |       '--allowedTools', agent.tools.join(','),

Branch Naming Convention

  main (or master)                       # Default branch
    |
    +--- commanddeck/{mission-id}/integration   # Integration branch per mission
    |       |
    |       +--- commanddeck/{mission-id}/obj-001    # Worker branch per objective
    |       |
    |       +--- commanddeck/{mission-id}/obj-002    # Parallel worker branch
    |       |
    |       +--- commanddeck/{mission-id}/obj-003    # Sequential phase 2 branch
    |
    |   (follow-up missions branch from integration, not main)
    |       +--- commanddeck/{follow-up-id}/obj-001
    |

  Worktree paths:
    ~/projects/{repo}                    # Main clone
    ~/projects/{repo}-wt-0               # Worker 0 worktree
    ~/projects/{repo}-wt-1               # Worker 1 worktree
    ~/projects/{repo}-wt-2               # Worker 2 worktree

Figure 8 — Branch naming and worktree path convention

Worktree Lifecycle

  executeBatch()
    |
    FOR each objective in batch:
    |
    +---> worktree.create(repo, index, branch, baseBranch)
    |       1. git worktree remove {path} --force (cleanup stale)
    |       2. rm -rf {path} (cleanup lingering dirs)
    |       3. ensureSshRemote() (HTTPS -> SSH conversion)
    |       4. git fetch origin
    |       5. git branch -D {branch} (delete stale local branch)
    |       6. git worktree add {path} -b {branch} {startPoint}
    |          startPoint = baseBranch || getDefaultBranch()
    |
    +---> worker.execute(wtPath, ...) [runs in worktree]
    |
    +---> worktree.remove(repo, index)
            1. git worktree remove {path} --force
            2. rm -rf {path}

Figure 9 — Worktree create/execute/remove cycle

Merge Strategy

  mergeCompleted()
    |
    +---> git checkout {integration_branch}
    |
    FOR each done objective with unmerged branch:
    |
    +---> git merge {obj.git_branch} --no-edit
    |       |
    |       +--- Success: obj.merged = true
    |       |
    |       +--- Conflict: invokeOBrien(branch)
    |               O'Brien reads briefings for intent
    |               Resolves conflicts, git add -A, git commit
    |               Runs tests to verify
    |               |
    |               +--- Success: obj.merged = true
    |               +--- Failure: git merge --abort, continue
    |
    +---> git checkout {default_branch}

Figure 10 — Integration branch merge with O'Brien conflict resolution

SSH Remote Conversion

worktree.ensureSshRemote() detects HTTPS origin URLs (https://github.com/...) and converts them to SSH (git@github.com:...). This is necessary because gh repo clone defaults to HTTPS, but the container authenticates via SSH keys.

Figure 13 — Evidence bundle schema with all fields

PR Body Generation

buildPRBody() assembles all evidence bundles into a structured PR body with:

  • Mission summary header with total objectives, phases, sessions, and file counts
  • Per-objective sections with file counts, test results, risk flags, and reviewer notes
  • Collapsible full evidence JSON section
  • HTML comments for Slack metadata: <!-- commanddeck:slack_channel=CXXXX --> and <!-- commanddeck:slack_thread_ts=1234.5678 -->

The Slack metadata comments enable GitHub Actions to post the UAT URL back to the correct Slack thread after deploying a preview environment.

SSE Streaming

The /api/telemetry/stream endpoint returns Server-Sent Events. Dashboard clients connect and receive real-time updates for span starts, trace completions, and log entries. A 30-second heartbeat keeps the connection alive.

Unix Socket for Worker Visibility

telemetry.js starts a Unix socket server at /tmp/commanddeck-telemetry.sock. The telemetry-emit.sh PostToolUse hook in worker sessions pipes tool call data to this socket via nc -U. This gives the dashboard real-time visibility into what tools each worker is calling.

19. Known Limitations

Honest assessment: These are real architectural limitations in the current system, not aspirational improvements.

Isolation

LimitationImpactCurrent Mitigation
No container isolation for agentsWorkers run as subprocesses with full host access. A misbehaving agent can read/write any file, access env vars, or interfere with other workers.--allowedTools restricts tool usage per agent, but this is advisory (CLI-enforced, not sandboxed).
--dangerously-skip-permissionsAll agents run without permission prompts. No human-in-the-loop for individual tool calls.Pre-write and pre-bash guard hooks block known dangerous patterns.
Shared filesystemAll agents share the same ~/.commanddeck directory and project clones. No namespace isolation.Worktrees provide branch isolation, but agents can read any worktree.

Communication

LimitationImpact
Slack-only channel supportNo WhatsApp, Telegram, Discord, or email integration. Slack is the sole interface.
Transactional Slack conversationsEach message is classified independently. No persistent conversation memory across sessions. Thread context is rebuilt by fetching history each time.
No streaming responsesSlack messages are posted as complete blocks. No partial/streaming updates to show agent progress in real-time.

Architecture

LimitationImpact
Raw claude -p instead of Agent SDKNo session resume, no native streaming, no agent teams. Each worker invocation is stateless. Context window is the entire prompt (no conversation history).
Single-process architectureCrash in any component takes down the entire system. No graceful degradation.
Env var secretsSLACK_BOT_TOKEN, SLACK_APP_TOKEN, GH_TOKEN are visible in the process environment. Any agent subprocess can read them.
No credential proxyAgents directly access GitHub via gh CLI with ambient credentials. No fine-grained token scoping per agent.
SQLite concurrencyWAL mode helps, but heavy parallel worker activity could cause lock contention. No connection pooling.
No task queueMissions run inline in the event loop. Long-running missions block the process. No retry or dead-letter queue.

Operational

LimitationImpact
No auto-recoveryIf the container crashes mid-mission, objectives may be left in in_progress state. Manual resume required via dashboard or API.
No backup strategySQLite databases and filesystem state have no automated backup. Loss of the EBS volume loses all state.
Single-tenant onlyNo user isolation, no multi-tenancy. One installation serves one team.
