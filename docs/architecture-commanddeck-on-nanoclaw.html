1. Executive Summary

CommandDeck is a self-hosted agentic software development platform: a user sends a Slack message, and a crew of specialized AI agents decomposes the task, implements it in parallel branches, runs tests, reviews for security and quality, creates a PR with structured evidence, deploys a preview environment, and waits for human approval to merge.

NanoClaw is an open-source agent orchestration framework (23.5k GitHub stars, Docker partnership, MIT license) that provides container-isolated agent execution, a credential proxy, filesystem-based IPC, multi-channel messaging (Slack, WhatsApp, Telegram, Discord, Gmail), and scheduled task execution.

The strategy: Fork NanoClaw and rebuild CommandDeck's development workflow on top of it. This gives us a better substrate (Agent SDK instead of raw claude -p, OS containers instead of worktrees-only, credential proxy, multi-channel support) while preserving CommandDeck's unique value: the mission lifecycle, agent crew, enterprise standards, risk-based mandatory reviews, evidence bundles, and the Slack approval flow.

What We Gain

  • Container isolation — Each agent runs in a Docker container with restricted filesystem mounts, replacing process-level worktree isolation
  • Credential proxy — Agents never see real API keys or tokens; a host-side HTTP proxy injects credentials transparently
  • Claude Agent SDK — Native query() with session resume, streaming, and agent teams, replacing raw subprocess management
  • Multi-channel — Slack, WhatsApp, Telegram, Discord, Gmail as first-class channels (Slack remains primary for approval flows)
  • Scheduled tasks — Built-in cron/interval/one-shot scheduling for health patrols, recurring reports, deployment checks
  • Active community — Upstream improvements (new container backends, security patches, channel integrations) can be cherry-picked

What We Preserve

  • The full mission lifecycle: decompose → parallel execute → merge → review → PR → deploy → approve
  • 11 specialized agent identities with enterprise standards and crew preferences
  • Evidence-based risk detection with mandatory specialist reviews
  • Git workflow: integration branches, parallel worktrees, O'Brien conflict resolution
  • The dashboard with traces, live logs, and metrics
  • Slack reaction-based approval flow for plan approval and PR merge/close

Architectural Principles

  1. NanoClaw is the substrate, CommandDeck is the product. NanoClaw handles container management, channel routing, scheduling, IPC, and credential management. CommandDeck adds the development-specific mission lifecycle, risk detection, evidence bundles, PR creation, and approval flows.
  2. MCP tools are the integration boundary. The mission lifecycle logic lives in the host orchestrator. Agents interact with it through custom MCP tools (create_worktree, detect_risks, submit_evidence, etc.) exposed via NanoClaw's IPC mechanism.
  3. Content is portable. Agent identities, standards, crew preferences, and playbooks are markdown files loaded via CLAUDE.md per NanoClaw group. They can be updated independently of code.
  4. Three databases, three concerns. NanoClaw's messages.db handles channel messages and scheduling. CommandDeck's app.db handles mission state. CommandDeck's telemetry.db handles observability. Separate failure domains.

Current vs. Target Isolation

Current CommandDeck

  • Git worktrees for file isolation
  • Same OS process namespace
  • Env vars for API keys
  • claude -p subprocess
  • Shared host filesystem
  • --allowedTools flag for tool restriction

Target (NanoClaw)

  • Docker containers for OS isolation
  • Separate PID/network/mount namespaces
  • Credential proxy (agents see placeholder keys)
  • Agent SDK query() with session resume
  • Explicit mount allowlist per agent
  • Per-group settings.json for tool config

New Mount: Worktree

The key extension to NanoClaw's mount system is the worktree mount. When a mission assigns an objective to an agent, the host:

  1. Creates a git worktree on the host: git worktree add {path} -b commanddeck/{mission}/{obj}
  2. Mounts the worktree read-write into the container at /workspace/worktree/
  3. The agent works in /workspace/worktree/ (not /workspace/group/)
  4. On completion, the host unmounts and manages the worktree (merge, cleanup)

This preserves CommandDeck's parallel-branch strategy while adding container isolation on top.

Container Image

NanoClaw's base image (node:22-slim + Chromium + Claude Code SDK) is extended with:

  • git and gh CLI (for agents that interact with git within their worktree)
  • Language runtimes as needed per project (configurable per group)
  • CommandDeck MCP tools pre-installed in the agent-runner

While NanoClaw treats all channels equally, CommandDeck extends Slack with approval flows:

  • Mission trigger: Messages in per-project channels (e.g., #weather-aggregator) trigger missions
  • Plan approval: Picard posts plan to Slack thread; user reacts with emoji to approve/reject
  • Status updates: Objective progress posted to thread as agents complete work
  • PR approval: PR URL posted to thread; reaction-based merge/close/iterate
  • UAT link: GH Actions deploys preview and posts URL back to Slack thread (reads metadata from PR body)
  • Meta-channel: #command-deck for project creation, onboarding, global commands

Secondary Channels

Non-Slack channels get notification-level access:

  • Mission started/completed/failed notifications
  • PR created/merged/closed notifications
  • Status queries ("what's the status of mission M-001?")
  • Future: full mission trigger and approval flows on any channel

Phase 1: Fork + Content Port

Goal: NanoClaw fork running locally with all 11 CommandDeck agents as groups.

  • Fork NanoClaw repository
  • Create groups/{agent}/CLAUDE.md for all 11 agents from current agents/*.md
  • Set up groups/global/CLAUDE.md with enterprise standards
  • Copy defaults/ (standards, crew preferences, playbooks)
  • Configure Slack channel via /add-slack skill
  • Test: send message, get routed to Picard, get decomposition

Phase 2: Mission Lifecycle

Goal: Full decompose → execute → merge → review → PR working locally.

  • Build src/commanddeck/mission.ts — port work loop from current lib/mission.js
  • Build src/commanddeck/ipc-tools.ts — handle CommandDeck MCP tools via IPC
  • Build src/commanddeck/worktree.ts — git worktree management, mount into containers
  • Build src/commanddeck/evidence.ts — evidence validation + PR body generation
  • Build src/commanddeck/risk.ts — risk detection from file patterns
  • Build src/commanddeck/pr.ts — PR creation via gh CLI on host
  • Build src/commanddeck/mission-db.ts — app.db schema and queries
  • Extend container-runner.ts for worktree mount support
  • Extend ipc.ts to route commanddeck_tool types to ipc-tools.ts
  • Test: full mission from Slack message to merged PR

Phase 3: Full Integration

Goal: Dashboard, telemetry, approval flows, health patrol all working.

  • Build src/commanddeck/dashboard.ts — HTTP server for dashboard + REST API
  • Build src/commanddeck/telemetry.ts — OTel SDK, telemetry.db, SSE
  • Build src/commanddeck/approval.ts — Slack reaction-based plan/PR approval
  • Port src/commanddeck/health.ts — register as NanoClaw scheduled task (2-min cron)
  • Port dashboard frontend (HTML/CSS/JS)
  • Wire up multi-channel notifications (WhatsApp, Telegram for status updates)
  • Test: full e2e with dashboard, traces, live logs, approval flow

Phase 4: Production Cutover

Goal: Replace deployed CommandDeck with new NanoClaw-based system.

  • Build production docker-compose.yml
  • Migrate existing app.db data (missions, objectives, threads)
  • Set up GH Actions CI/CD for fork (test → build → deploy)
  • Deploy to EC2 alongside existing CommandDeck
  • Cut over Slack to new instance
  • Verify all onboarded projects work (WeatherAggregator, FurEverCare, etc.)
  • Decommission old CommandDeck deployment
Risk mitigation: Keep the old CommandDeck running on EC2 throughout migration. The new system uses different database files and container names — they can coexist. Only cut over Slack routing once the new system passes e2e validation with a test project.
