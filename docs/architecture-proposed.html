1. Executive Summary

CommandDeck is a self-hosted agentic software development platform: a user sends a Slack message, and a crew of specialized AI agents decomposes the task, implements it in parallel branches, runs tests, reviews for security and quality, creates a PR with structured evidence, deploys a preview environment, and waits for human approval to merge.

The problem: CommandDeck currently operates in a transactional mode. Every Slack message is a one-shot classification — Picard reads it, classifies it, and either launches a mission or sends a single response. There is no memory between messages. The user cannot brainstorm with Picard, refine an approach, or ask follow-up questions outside of an active mission. Additionally, workers run as raw claude -p subprocesses sharing the host's environment with no container isolation or credential protection.

The strategy: Upgrade CommandDeck in place with four targeted changes. No fork, no rewrite, no new framework dependency. The entire mission lifecycle remains untouched.

The Four Upgrades

# Change What It Does Scope

2 Persistent Sessions Every Slack thread becomes a persistent Picard conversation with memory. Mission creation becomes a tool call, not a classification. lib/worker.js new converse() method + lib/thread.js session tracking — ~200 lines new
3 Container Isolation Implementation workers (Borg, Redshirts, etc.) run in Docker containers with restricted mounts lib/container.js new module — ~150 lines new
4 Credential Proxy Lightweight HTTP proxy on localhost:3001; containers get placeholder keys, proxy injects real credentials lib/credential-proxy.js new module — ~120 lines new

What We Preserve

  • The full mission lifecycle: decompose → parallel execute → merge → review → PR → deploy → approve (lib/mission.js — 907 lines unchanged)
  • Evidence-based risk detection with mandatory specialist reviews (lib/risk.js — 133 lines unchanged)
  • Evidence bundle validation and PR body generation (lib/evidence.js — 169 lines unchanged)
  • PR creation, merge, close, and branch cleanup (lib/pr.js — 194 lines unchanged)
  • Git workflow: worktrees, integration branches, O'Brien conflict resolution (lib/worktree.js — 186 lines unchanged)
  • Health patrol: stuck detection, thrashing, test failure loops (lib/health.js — 250 lines unchanged)
  • Dashboard + telemetry: traces, logs, metrics, SSE streaming
  • Database schema: app.db + telemetry.db (one column addition to threads table)
  • All 11 agent identities from agents/*.md
  • Hooks system, configuration, standards, crew preferences, playbooks
Key metric: ~500 lines of new code, 3 new files, 2 modified files, 0 lines of mission lifecycle rewritten. The entire upgrade is contained in the worker layer and the Slack-facing conversation layer. The orchestration engine is untouched.

Architectural Principles

  1. Upgrade, don't rewrite. Every new component is additive. The mission lifecycle code (mission.js, 907 lines) is not modified at all. New functionality wraps around the existing system.
  2. The conversation is the interface. Persistent Picard sessions replace transactional classification. Mission creation becomes a tool call from within a conversation, not a separate code path.
  3. Isolation is layered. Containers add OS-level isolation on top of existing worktree-based file isolation. The worktree system is preserved; containers are wrapped around it.
  4. Credentials stay on the host. The credential proxy is the single point where real API keys exist. Containers see only placeholders. The .env file is never mounted.

3. The Conversational Model

This is the headline feature of CommandDeck v2. Today, every Slack message is a transaction: Picard reads it, classifies it (work / converse / inquiry / onboard / create), and either launches a mission or sends a single response with no memory. Tomorrow, every Slack thread becomes a persistent conversation with Captain Picard who has full context of everything discussed.

The shift: Mission creation moves from being an action the system takes on a classified message to being a tool call Picard makes during a conversation. Picard decides when the user's intent is clear enough to launch work — and by that point, the full conversation context flows into the decomposition, producing better plans.

Current vs. Proposed Flow

Current: Transactional

  1. User sends message
  2. Thread.js debounces (4s)
  3. classifyMessage() — one-shot claude -p
  4. Returns JSON: {action, message, task_description}
  5. If action=work: launch mission immediately
  6. If action=converse: send single response, forget
  7. Next message starts over from scratch

No conversation memory. Every message is independent.

Proposed: Conversational

  1. User sends message
  2. Thread.js debounces (4s)
  3. worker.converse() — persistent Agent SDK session
  4. Picard responds conversationally (full thread context)
  5. User and Picard go back and forth (N messages)
  6. When ready, Picard calls create_mission tool
  7. Mission launches with full conversation context
  8. Conversation continues — status updates, follow-ups

Full memory. Picard remembers everything.

The create_mission Tool

This is the bridge between conversation and execution. When Picard determines the user's intent is clear and actionable, Picard calls this tool within the conversation session.

Aspect Details
Receives { repo: string, description: string } — Picard writes the description based on the full conversation context. The description is richer than what a user would type in a single message because Picard has already discussed the approach, constraints, and preferences.
Context Available Picard's session has the complete conversation history. When decompose() runs, the agent prompt includes the enriched description. Picard already knows the approach because it was discussed during the conversation.
Returns { mission_id: string, status: "planning" } — returned to the conversation so Picard can tell the user the mission has been launched and what to expect.
Side Effects Calls Mission.start() (existing code, unchanged). Updates thread row: mission_id, status='executing'. Posts plan to Slack thread via existing reporter.
Error Handling If mission creation fails, the error is returned to Picard's session. Picard can explain the issue to the user and suggest fixes. The conversation continues.
